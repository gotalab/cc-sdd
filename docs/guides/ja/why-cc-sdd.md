# cc-sdd という賭け (philosophy note)

> [English](../why-cc-sdd.md) | 日本語

これは「cc-sdd がなぜ存在し、どんなトレードオフを取っているのか」の長い版である。install して試したいだけなら、[プロジェクトの README](../../../README.md) の方が早い。設計の根拠を理解したいときに、ここに来てほしい。

## 短い版

cc-sdd は spec を、システムの各部分の間の契約として扱う。エージェントに手渡す「命令書」ではない。コードは依然として source of truth で、spec はコード各部分の間の契約を明示化するために使う。そうすることで、人間とエージェントが常に同期を取らなくても並列に動けるようになる。

賭けはシンプルだ。適切な粒度で明示化された契約こそが、チーム規模の AI 駆動開発を速くする。遅くしない。境界はオーバーヘッドではなく、境界があるから自由に動ける。

## Specification と Design を分ける

しばしば混同される 2 つを分けると分かりやすい。

- **Specification**: 契約、境界、その仕事が尊重すべき前提条件。
- **Design**: その契約の *内側* にある自由な探索空間。コンポーネント、インターフェース、実装判断。

cc-sdd はこの区別を一次情報として扱う。

- `requirements.md`、File Structure Plan、境界アノテーション (`_Boundary:_`、`_Depends:_`) が **specification** を定義する。守るべき契約である。
- `design.md` の内部、`tasks.md` の順序付け、各 task 内部の実装は **design** である。契約の内側で、エージェントが自由に動ける領域である。

人間は specification レイヤーでレビュー・承認する。エージェントは design レイヤーの内側で自由に動く。

## cc-sdd における「速さ」の考え方

spec は開発を遅くする、というのが一般的な前提である。cc-sdd は逆の前提に立つ。適切な粒度で明示化された契約こそが、チーム規模の AI 駆動開発を速くする。そこから 4 つのことが導かれる。

### monolith ではなく right-sized spec

spec は数時間から数日で出荷できる単位に収まるべきである。1 spec に収まらない規模の仕事は `/kiro-discovery` が分解し、`/kiro-spec-batch` が複数 spec を並列生成する。一つの巨大計画にコミットするのではなく、一つ出して、学んで、次に進む、を繰り返す。

### 連邦的な責務分担

各 spec は自分のスコープとオーナーを持つ。cross-spec review が spec 間の不整合を拾うので、中央の権威が master plan を握る必要はなく、仕事は明示的な契約を通じて連携する。

### 継続的な機械的検証

契約はテスト、lint、independent reviewer pass、境界チェック、`/kiro-validate-impl` によって開発中ずっと検証される。最後の統合工程でまとめて確認するのではない。

### エージェントが spec を書き、人間は契約をレビューする

各フェーズはエージェント駆動である。`requirements.md` も `design.md` も `tasks.md` も、自分で書くわけではない。ただし phase gate で markdown を読むコストはゼロではなく、実際に読むことになる。重要なのは、そのレビューの焦点が契約レベルの判断（境界、責務、依存関係のスコープが正しいか）に絞られており、行単位の挙動記述をトレースする作業ではないことだ。タスク完了時には、承認済み契約に対して実際の `git diff` をレビューする。phase gate レビューがあるからこそ、契約の問題が統合時ではなく review 時に表面化する。

## なぜ AI エージェントを使う文脈で重要なのか

エージェントは速い。1 つのセッションで、複数のモジュールにわたって何千行ものコードを平気で並列生成する。ボトルネックは能力ではない。coordination (調整) である。

エージェントがある spec で module A を変更し、別の spec で module B を変更しているとき、A と B の間の契約がどこにも書かれていなければ、silent breakage は後工程まで表面化しない。これは大規模エンジニアリングチームが何十年も、明示的なインターフェース、contract test、モジュール境界で解決してきた痛みそのものである。cc-sdd はその原則をエージェント駆動の開発に適用している。

境界の内側ではリファクタリング、発明、反復を自由にやっていい。境界の外側には契約があるから、あなたが動いても他の人の仕事が壊れない。

## cc-sdd が向いている場面

- 仕事が複数 spec の中粒度の出荷に分解される。一枚岩でもなく、単発の行変更でもない。
- 複数の人間・エージェント・ストリームがコードベースに触れていて、「あなたの変更、私のに影響ある？」が時間を食い始めている。
- 小さい垂直スライスで出荷して、各スライスから学んでから次にコミットしたい。
- エージェント生成コードの任意の行を、承認済み契約までたどれる監査性が必要。

## cc-sdd が不要な場面

- 1 エージェント 1 セッションで収まる個人作業。
- 境界を書き下す必要がないプロトタイプ、使い捨てコード。
- 契約を明示化するより「vibe coding」の方が本当に速い仕事。

cc-sdd を使う場合でも、`/kiro-discovery` は *「spec 不要、直接実装して OK」* を valid な出力として返せる。全ての変更に formal な spec を付けようとしているわけではなく、システム各部分間の契約が実際にコストに見合うケースだけを対象にしている。

> ディシプリンがオーバーヘッドに感じるなら、多分 spec が大きすぎる。もっと小さく分解しよう。
