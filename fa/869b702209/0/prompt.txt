  アプローチ提案

  アプローチ A: ボトムアップ（データモデル中心）

  どう作るか: まずSQLiteのブロック・ページ・データベースのスキーマを堅牢に設計し、その上にCRUD API → React UI
  の順で積み上げる。TipTapやProseMirrorなどの既存エディタフレームワークは使わず、自前ブロックモデルで構築。

  - Pros: データ構造が最適化される、CLI連携が自然、将来の拡張に強い
  - Cons: エディタ体験の品質を自力で確保する必要がある、開発量が多い
  - 規模: 特大

  アプローチ B: エディタフレームワーク活用（推奨）

  どう作るか: BlockNoteやNovel（TipTap/ProseMirror系）をブロックエディタの基盤に採用し、Notionライクなスラッシュコマンド・ドラッグ&ドロップ・インラインスタイルを高速に実現。ページ管理
  ・データベース・CLIは自前で構築。SQLiteはメインプロセス側でbetter-sqlite3で管理し、IPCでレンダラーと通信。

  - Pros: エディタ体験が最初から高品質、Notionに近いUX を最短で実現、スラッシュコマンド/D&D/インラインスタイルが組み込み済み
  - Cons: フレームワークの制約に合わせる場面がある、カスタムブロック追加にはフレームワーク理解が必要
  - 規模: 大（ただしAより大幅に少ない）

  アプローチ C: 既存OSSフォーク

  どう作るか: AFFiNE、AppFlowy等のオープンソースNotionクローンをフォークして、SQLiteローカル完結に改造する。

  - Pros: 完成度の高いUIが最初から手に入る
  - Cons: 他人のアーキテクチャに縛られる、フォーク元との差分管理が困難、学びが少ない                                                                                                     - 規模: 中（ただし改造コストは予測困難）
                                                                                                                                                                                         ---
  推奨: アプローチ B

  理由：ブロックエディタの品質がNotionの体験の8割を決めます。BlockNoteはNotionクローン用途に設計されており、スラッシュコマンド・D&D・インラインスタイルが最初から揃っています。残りのペ
  ージ管理・データベース・CLI・サイドバーは自前で作り込むことで、自由度とNotionらしさを両立できます。

下記は参考になる？
結論から言うと、cc-sdd 3.0 の勝ち筋は「全部入りの agent OS」ではなく、**approved spec を executable work に変える、skills-first の軽量ハーネス**です。PR #141 は、もともとの cc-sdd の強みだった `Requirements → Design → Tasks` の規律に、**native subagent-driven implementation、独立 reviewer、feature-level final validation、brainstorm/brief/roadmap による前段整理**を足していて、単なる「仕様書生成器」から「仕様駆動の実行基盤」に寄っています。外部の比較でも、3月時点の cc-sdd は「要件→設計→タスクの規律」が強みで、レビュー工程や一気通貫の実装導線は前面に出ていないと見られていましたが、PR #141 はまさにその弱点を埋める方向です。 ([GitHub][1])

いま一番強く押せるのは、**「少ない部品で end-to-end がつながる」こと**です。PR では外部の Ralph Loop 依存を落として `kiro-impl` に統合し、タスクごとに fresh subagent を立て、reviewer は実装者の自己申告ではなく実際の `git diff` を読む形に変わっています。さらに controller を軽く保つため、Task Brief の具体化や diff 読み取りを subagent 側に寄せ、`NEEDS_CONTEXT / BLOCKED / DONE_WITH_CONCERNS` のような正直な状態遷移も入っています。これは派手ではないですが、**長時間走るときに壊れにくい設計**としてかなり強いです。 ([GitHub][2])

もう一つの強みは、**「変更容易性が高い」ことを実装で証明している**点です。共有ルールを skill ディレクトリに物理コピーせず、`metadata.shared-rules` で install 時解決に変えたことで、agent×skill の重複を減らして single source of truth に寄せています。これは README に書いてある「Customize once」に実体を与える変更で、あなたが言う「シンプルで不要物を減らす」「変えやすい」にかなり一致しています。競合が大きい surface area を持つほど、ここは差別化しやすいです。 ([GitHub][2])

さらに、**spec を“文書”ではなく“制御面”にしている**のも売りやすいです。PR では design に File Structure Plan を追加し、それが task の `_Boundary_` や Task Brief に流れ、brainstorm は `brief.md` と `roadmap.md` を書いて次の skill に文脈を受け渡し、`spec-batch` では multi-spec を dependency wave でさばけるようになっています。つまり cc-sdd 3.0 は「requirements/design/tasks を作る」だけではなく、**その artifact が次工程の挙動を直接決める**構造になっています。ここはかなり本質的です。 ([GitHub][2])

他の harness と並べると、立ち位置はかなり明確です。Everything Claude Code は、skills・instincts・memory optimization・security scanning・MCP・legacy shims まで含む大きなシステムで、2026年4月の公開面では 38 agents / 156 skills / 72 shims まで広がっていますし、Claude Code 側の制約で rules を手動インストールする手順も残っています。Superpowers は、auto-trigger される skills、worktree、TDD、subagent-driven-development まで含めた**完全な開発方法論**として打ち出しています。Claude Harness は 5 verbs と TypeScript guardrail engine を中心に、Plan→Work→Review→Release の**崩れにくい標準フロー**を主張しています。なので cc-sdd は、ここに「全部ある」「最強ガードがある」で戦うより、**最小 surface / spec-native / customize-once / cross-agent portability**で勝つのが自然です。 ([GitHub][3])

なので売り文句は、次の軸が合っています。
**1. Specs you can run.**
**2. The lightweight harness for approved work.**
**3. Kiro-compatible SDD rebuilt as portable skills.**

日本語なら、
**「仕様書を、読むものから実行するものへ。」**
**「承認済み requirements / design を、実装・レビュー・最終検証までつなぐ最小ハーネス。」**
**「全部入りではなく、壊れにくく変えやすい spec-to-implementation 基盤。」**
あたりが強いです。

README や LP の一行目も、今なら **“long-running autonomous implementation” 単体** より、**“approved specs become executable work”** を前に出すほうが差が伝わります。自律実装は競合も言いますが、「approved spec が実行可能な単位になる」は cc-sdd の artifact 設計とかなり噛み合っています。しかも current branch README 自体もその方向に寄せています。 ([GitHub][4])

2026/4 以降に通用するかで言うと、**かなり通用する**と思います。理由は、agent ecosystem が skills を中心に収束しつつあるからです。Anthropic は Claude Code で custom commands を skills に統合し、skills は open Agent Skills standard に従い、必要時だけ読み込まれ、subagent execution と組み合わせられると説明しています。OpenAI も Codex で skills を reusable workflow の authoring format と位置づけ、CLI/IDE/app で使え、metadata だけ先に見せて必要時に `SKILL.md` を読む progressive disclosure を公式化しています。OpenAI API 側も skills を versioned bundle + `SKILL.md` manifest と定義し、open Agent Skills standard 互換としています。Warp も skills を agent として実行でき、ローカルでは `.claude/skills/`、`.codex/skills/`、`.cursor/skills/` などを自動 discovery し、rules は `AGENTS.md` を使います。つまり **skills-first / repo-native artifact / AGENTS.md 系の設計は、一過性ではなく流れに乗っている**と言えます。 ([Claude API Docs][5])

しかも Codex では、`~/.codex/prompts` の custom prompts が 0.117.0 で消え、cc-sdd 側でも `--codex` prompts mode を deprecated / blocked にして `--codex-skills` へ寄せています。これは単なる互換対応ではなく、**“prompt hacks から skills へ” という市場移行に正面から乗れている**ということです。今このタイミングで 3.0 を skills 中心に振るのは、かなり筋がいいです。 ([GitHub][6])

ただし注意点も一つあります。Codex では、custom prompts の代替として skills に移った結果、**implicit invocation が効きすぎるのでは**という不満も出ています。実際、explicit-only skill injection を求める issue もあり、旧 prompt UX と完全同等ではないという声があります。なので cc-sdd 3.0 は、skills-first を押しつつも、**明示的な `/kiro:*` / `$kiro-*` 導線を分かりやすく保つ**、description を狭く書いて誤発火しにくくする、必要なら manual-first な導入プロファイルを用意する、の3つをやるとかなり安定します。 ([GitHub][7])

要するに、売り出し方はこれです。
**「cc-sdd 3.0 は、全部入り harness ではない。承認済みの spec を、Claude Code / Codex / 次の skills-capable agents でも動く executable workflow に変える、最小で変えやすい SDD harness だ。」**

## Detailed Findings: GitHub Copilot Agent Skills Specification & Conventions

Based on the latest documentation and research, here's what I found:

### 1. **Skills Directory Structure**

Skills are stored in **project-level or personal-level directories**:

**Project-level (repository):**
- `.github/skills/`
- `.claude/skills/`
- `.agents/skills/`

**Personal-level (home directory):**
- `~/.copilot/skills/`
- `~/.claude/skills/`
- `~/.agents/skills/`

Each skill requires its own **subdirectory containing a `SKILL.md` file**. Example: `.github/skills/my-skill/SKILL.md`

---

### 2. **SKILL.md Frontmatter Specification**

The SKILL.md file uses **YAML frontmatter followed by Markdown content**.

**Required fields:**
- `name` (string): Unique, lowercase identifier using hyphens for spaces (e.g., `my-skill`)
- `description` (string): Explains what the skill does and when Copilot should use it (max 1024 characters)

**Optional fields:**
- `license`: Description of applicable licensing
- `allowed-tools`: Pre-approved tools (array) like `["shell"]` — requires careful review as it removes confirmation prompts
- `argument-hint`: Hints for users on how to invoke the skill
- `user-invocable`: Boolean to control whether users can manually select the skill
- `disable-model-invocation`: Boolean to require manual invocation only (prevents automatic selection by Copilot)

**Example structure:**
```yaml
---
name: my-skill
description: Explains what the skill does and when to use it
allowed-tools: ["shell"]
user-invocable: true
---
# My Skill

Detailed instructions, examples, and guidelines for Copilot to follow...
```

---

### 3. **Skill Invocation Methods**

**Automatic invocation (default):**
- Copilot automatically determines when to use skills based on the user's prompt and the skill's description
- The SKILL.md is injected into the agent's context when relevant

**Manual/slash command invocation:**
- Users type `/` in chat to view available skills
- Select from the list (e.g., `/my-skill`)
- Can add context after (e.g., `/my-skill for the login page`)
- Requires `user-invocable: true` in frontmatter (default behavior)

**Disable automatic invocation:**
- Set `disable-model-invocation: true` to require manual invocation only

---

### 4. **Multi-Agent & Sub-Agent Support**

GitHub Copilot **does support multi-agent orchestration** through:

**Fleet Command (Copilot CLI):**
- Use `/fleet` in Copilot CLI to run multiple agents in parallel
- Behind-the-scenes orchestrator decomposes tasks into discrete work items with dependencies
- Dispatches independent sub-agents simultaneously
- Each sub-agent gets its own context window but shares the filesystem

**Custom Agents & SDK Integration:**
- Copilot SDK enables defining custom agents with specific expertise
- When user request matches an agent's description, runtime **automatically delegates** as a sub-agent
- Sub-agents run in isolated context with scoped tools
- Parent session receives lifecycle events (`subagent.started`, `subagent.completed`, `subagent.failed`)

**Custom Agent Definition Format (SDK):**
```typescript
{
    name: "agent-id",                    // unique identifier
    displayName: "Human-Readable Name",  // user-facing label
    description: "What the agent does",  // guides intent-matching
    tools: ["tool1", "tool2"],           // accessible tools
    prompt: "System instructions",       // behavior/expertise
    mcpServers: { /* optional */ },      // custom MCP servers
    infer: true                          // auto-selection toggle
}
```

---

### 5. **Agent Skills Open Standard**

The Agent Skills specification is an **open standard maintained by Anthropic**, hosted at [agentskills.io](https://agentskills.io) with a formal specification at [agentskills.io/specification](https://agentskills.io/specification).

**Key characteristics:**
- Simple, open format for giving agents new capabilities
- Skills are folders containing instructions, scripts, and resources
- Used by multiple AI systems (GitHub Copilot, Claude, etc.)
- Community contributions welcome
- Reference SDK and example implementations available

---

### 6. **Custom Agent Definitions** (Beyond Skills)

Custom agents support **YAML frontmatter + Markdown**:

**Supported frontmatter fields:**
- `name`: Display name for the agent
- `description` (required): Explains agent's purpose and capabilities
- `target`: Environment (`vscode` or `github-copilot`); defaults to both
- `tools`: List of available tools; omitting enables all tools
- `model`: Specifies which model the agent uses
- `disable-model-invocation`: Prevents automatic selection; requires manual invocation
- `user-invocable`: Controls manual agent selection
- `mcp-servers`: Configures additional MCP servers and tools
- `metadata`: Key-value pairs for annotations

**Markdown body** (max 30,000 characters): Defines behavior, expertise, and instructions.

| Field | Type | Requirements | Notes |

- **compatibility** (string): List required tools, libraries, or dependencies (rarely used)

**Sequential vs. Parallel:**
- Cascade handles complex tasks through sequential multi-step **Flows** (not parallel agents)

### 2. **SKILL.md Frontmatter Specification**

SKILL.md uses **YAML frontmatter** (enclosed in `---` markers) with the following fields:

**Required fields:**
- **`name`** (1-64 characters)
  - Lowercase letters, numbers, hyphens only
  - Must match parent directory name
  - Cannot contain XML tags
  - Cannot contain reserved words: "anthropic", "claude"

  - **CRITICAL for routing** - Claude uses this to decide when to trigger the skill
  - Should state what the skill does AND when to use it
  - Include specific trigger keywords
  - Include negative boundaries ("Do NOT use for...")
  - Write in third person (not first/second person)
  - Example: "Extract text and tables from PDF files, fill forms, merge documents. Use when working with PDF files or when the user mentions PDFs, forms, or document extraction."

**Optional fields:**
- **`disable-model-invocation`** (boolean, default: false)

- **`user-invocable`** (boolean, default: true)
  - Controls whether skill appears in the `/` slash command menu

- **`argument-hint`** (string)
  - Hint text displayed for slash command usage

- **`context`** (string, values: "fork" or "inline")
  - `fork`: Spawns isolated sub-agent with separate context window
  - `inline`: Executes in current agent's context

- **`metadata`** (object/key-value map)
  - Optional catch-all for version numbers, tags, etc.
  - Example: `metadata: { version: "1.0.0", category: "document-processing" }`

- **`license`** (string)
  - Signals trust/provenance: `license: Proprietary. LICENSE.txt has complete terms`

- **`allowed-tools`** (array, experimental)
  - Declares which tools the skill needs for permission scoping

**Example frontmatter:**

  1. reviewer 指摘を「本当に外部依存の blocker か」「この repo 内で補える前提不足か」に分解する
  2. package.json、依存解決、起動コマンド、native module ABI を先に調査する
  3. repo 内で補えると判明した時点で blocked を書かず、そのまま修正ラウンドを続ける

  - repo 内で追加できる依存や設定なら blocker にしない

  - native module があるときは Node/Electron の両実行系を最初に分けて考える

  - → format: 'es' に変更 + await import('electron-store') で解決

  タスク名: "Test dynamic import of electron-store in Electron", "Test if the Electron app launches with ESM output"

  2. better-sqlite3 のネイティブモジュール問題

  - better-sqlite3 は C++ ネイティブアドオン。Node.js のバージョンに紐づくバイナリが必要
  - システムの Node.js と Electron 内蔵の Node.js でABIバージョンが異なる
  - ユニットテスト用にビルドしたバイナリでは Electron で動かない（逆も然り）
  - → electron-rebuild で Electron 用に再ビルドが必要

  タスク名: "Test better-sqlite3 in Electron", "Test Electron with @core/data-model import"

  3. ウィンドウが表示されない連鎖障害

  上記2つが原因で CoreBridge の初期化が失敗 → ウィンドウが作られない → Playwright が firstWindow() でタイムアウト

  タスク名: "Debug why window isn't created", "Debug Playwright Electron - check window creation" (×4回ほど試行)

  - JSX transform: electron-vite/Vite 7 が classic JSX を使うため、全 .tsx に import React が必要だった
  - Preload パスが .js → .mjs に変わった
  - sandbox: false が ESM preload に必要だった
