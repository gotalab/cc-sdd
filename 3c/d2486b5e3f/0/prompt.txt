  アプローチ提案

  アプローチ A: ボトムアップ（データモデル中心）

  どう作るか: まずSQLiteのブロック・ページ・データベースのスキーマを堅牢に設計し、その上にCRUD API → React UI
  の順で積み上げる。TipTapやProseMirrorなどの既存エディタフレームワークは使わず、自前ブロックモデルで構築。

  - Pros: データ構造が最適化される、CLI連携が自然、将来の拡張に強い
  - Cons: エディタ体験の品質を自力で確保する必要がある、開発量が多い
  - 規模: 特大

  アプローチ B: エディタフレームワーク活用（推奨）

  どう作るか: BlockNoteやNovel（TipTap/ProseMirror系）をブロックエディタの基盤に採用し、Notionライクなスラッシュコマンド・ドラッグ&ドロップ・インラインスタイルを高速に実現。ページ管理
  ・データベース・CLIは自前で構築。SQLiteはメインプロセス側でbetter-sqlite3で管理し、IPCでレンダラーと通信。

  - Pros: エディタ体験が最初から高品質、Notionに近いUX を最短で実現、スラッシュコマンド/D&D/インラインスタイルが組み込み済み
  - Cons: フレームワークの制約に合わせる場面がある、カスタムブロック追加にはフレームワーク理解が必要
  - 規模: 大（ただしAより大幅に少ない）

  アプローチ C: 既存OSSフォーク

  どう作るか: AFFiNE、AppFlowy等のオープンソースNotionクローンをフォークして、SQLiteローカル完結に改造する。

  - Pros: 完成度の高いUIが最初から手に入る
  - Cons: 他人のアーキテクチャに縛られる、フォーク元との差分管理が困難、学びが少ない                                                                                                     - 規模: 中（ただし改造コストは予測困難）
                                                                                                                                                                                         ---
  推奨: アプローチ B

  理由：ブロックエディタの品質がNotionの体験の8割を決めます。BlockNoteはNotionクローン用途に設計されており、スラッシュコマンド・D&D・インラインスタイルが最初から揃っています。残りのペ
  ージ管理・データベース・CLI・サイドバーは自前で作り込むことで、自由度とNotionらしさを両立できます。
